# Memory Bank

This folder is our shared project memory.
It helps keep the project goals and current understanding in one place so both the user and the agent can stay aligned.

## Files

- `goals.md`: the high-level project goals and success criteria
- `product-spec-v1.md`: the version 1 product specification
- `information-architecture-v1.md`: the version 1 navigation and screen structure
- `authentication-spec-v1.md`: the detailed authentication requirements for version 1
- `non-functional-requirements-v1.md`: implementation constraints for performance, security, reliability, accessibility, and responsiveness
- `ui-ux-design-system-v1.md`: design principles, visual system, components, and detailed UI flows
- `frontend-architecture-v1.md`: frontend folder structure, state strategy, and recommended libraries
- `backend-architecture-v1.md`: backend API specification, schema, layers, and cross-cutting concerns
- `validation-rules-v1.md`: domain and form validation rules for core finance modules
- `notifications-alerts-v1.md`: notification examples and delivery channels for version 1
- `acceptance-criteria-v1.md`: concise functional definition-of-done criteria for core V1 modules
- `high-level-design-v1.md`: system-level design, module boundaries, architecture direction, and runtime guidance
- `low-level-design-v1.md`: implementation-oriented design for frontend, backend, data, security, and jobs
- `implementation-roadmap-v1.md`: phased execution roadmap from foundation through deployment readiness
- `repo-structure-starter-scaffolding-v1.md`: recommended repository layout and starter scaffolding plan
- `platform-deployment-options.md`: platform, container, identity, observability, and deployment choices for current and future phases
- `future-enhancements.md`: later roadmap ideas that are intentionally outside current V1 scope
- `current-context.md`: the latest agreed understanding, decisions, and open questions

## How We Use This

1. Update `goals.md` when the project direction or priorities change.
2. Capture the current version scope in `product-spec-v1.md`.
3. Capture the app structure and navigation model in `information-architecture-v1.md`.
4. Capture focused feature details such as authentication in dedicated spec documents.
5. Capture implementation constraints in `non-functional-requirements-v1.md`.
6. Capture design rules and flows in `ui-ux-design-system-v1.md`.
7. Capture frontend implementation structure in `frontend-architecture-v1.md`.
8. Capture backend API and persistence structure in `backend-architecture-v1.md`.
9. Capture validation requirements in `validation-rules-v1.md`.
10. Capture notification and alert behavior in `notifications-alerts-v1.md`.
11. Capture acceptance criteria in `acceptance-criteria-v1.md`.
12. Capture system-level design in `high-level-design-v1.md`.
13. Capture implementation-level design in `low-level-design-v1.md`.
14. Capture implementation sequencing in `implementation-roadmap-v1.md`.
15. Capture repo and scaffolding decisions in `repo-structure-starter-scaffolding-v1.md`.
16. Capture platform and deployment choices in `platform-deployment-options.md`.
17. Capture later roadmap ideas in `future-enhancements.md`.
18. Capture the latest shared understanding in `current-context.md`.
19. Keep entries short, dated, and easy to scan.

## Update Rules

- Prefer appending new entries instead of deleting history.
- Keep goals stable, but refine them when the project direction becomes clearer.
- Use this folder as the source of truth before starting major work.
